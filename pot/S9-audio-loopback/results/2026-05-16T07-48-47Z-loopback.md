## G3 — two-way audio

Self-attested by operator: heard audio in both directions, no dropouts. Codec: PCMU (ulaw 8 kHz).

The audio path required forcing pjsua to use the ALSA `pulse` device (index 6 in pjsua's enumeration on this host) via `--capture-dev=6 --playback-dev=6` in both account configs — see commit `4af8d06`. Without that, the second pjsua TUI hit ALSA EBUSY on `/dev/snd/pcmC0D0p` and silently fell back to no audio device. Echo (extension 600) was unaffected because only one pjsua ran for that probe.

## Linux + Docker Desktop quirks captured along the way

Three real bugs surfaced during execution. Each shipped its own commit; readout records them as future-operator triage hints.

1. **Bind-mount denial** (`mounts denied: path is not shared from the host`). Docker Desktop on Linux only forwards a fixed allow-list of host paths; `/media` isn't on it by default. Fix: bake configs into the image via `COPY` in the Asterisk Dockerfile (commit `d28b1e9`).

2. **`local_net` + Docker bridge overlap**. The original `pjsip.conf` had `local_net=172.16.0.0/12`, which covers the Docker bridge subnet (default-assigned anywhere in 172.16/12). PJSIP read that as "treat the softphone as local, skip `external_media_address` rewriting" — so SDP advertised the container's own bridge IP, unreachable from the host softphone. Calls connected, dialplan ran, zero audio. Fix: drop all `local_net` directives — every peer is external in the Docker-Desktop loopback topology (commit `9814992`).

3. **pjsua's default device locks `/dev/snd/pcmC0D0p` exclusively**. Two pjsua TUIs on the same Linux host can't both use audio because the auto-selected `lavrate` ALSA plug terminates at the hardware device. Manifests as: signalling fine, INVITE/200 OK/ACK clean, but bob's pjsua log floods with `snd_pcm_hw_open ... -16: Device or resource busy` at every fallback rate, RTP TX count = 0, no sound either way. The bug looked like an Asterisk bridge-tech failure (`simple_bridge` upgraded to `native_rtp` mid-call) until the bob pjsua log was inspected. Fix: pin both account configs to `--capture-dev=6 --playback-dev=6` (the `pulse` plug → PipeWire mixer → no contention) (commit `4af8d06`).

## What this readout does NOT validate

- **macOS path.** Spike was originally specified for macOS; only Linux was tested in this run. Three of the three quirks above are Linux-specific (the `local_net` bug would manifest similarly on Mac Docker Desktop, but the bridge subnet is different; pjsua device 6 is Linux-only — macOS lacks the `pulse` ALSA plug). The Mac path still needs an operator run; an audit-trail readout under the same `results/` directory will let the two stack up.
- **Kamailio-fronted SBC topology.** Deferred per ADR-0025; the S1 PoT spike retains the carrier-trunk evidence under the alternate topology.
- **High concurrent call counts.** RTP port range is 10000–10010 (11 ports = ~5 concurrent calls). Bench scale only.
- **TLS / WSS / DTLS-SRTP.** UDP-only per S9 hypothesis.

**Green** — 2026-05-16, single-sitting G1–G5 on Linux + Docker Desktop. Validates ADR-0025 audio path. Supersedes the S1 Layer-2 rtpengine media-smoke carry-over for MVP audio evidence (Kamailio + rtpengine paths remain documented but unrequired under ADR-0025).
